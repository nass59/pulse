# KRaft mode

> **Why you'll hit this:** the very first time you run `docker compose up`, Kafka isn't ready for a few seconds — and you'll wonder why. KRaft is the reason.

Picture a small club. Someone has to keep the membership roll: who's in, who's in charge, what rooms exist. The old Kafka hired a **separate clerk** for that job — a whole **ZooKeeper** ensemble running alongside the brokers, just to hold the cluster's metadata. Two distributed systems to run and babysit, not one.

**KRaft** (Kafka Raft) fires the clerk and lets the brokers keep their own roll. The **controller** — the node responsible for cluster metadata and leader election — now lives *inside* a broker process, backed by an internal Raft-replicated log. One fewer thing to deploy, secure, and reason about. Pulse runs KRaft from day one; ZooKeeper never enters the picture.

In `infra/docker-compose.yaml` the broker is told to play both parts:

```yaml
KAFKA_PROCESS_ROLES: broker,controller
KAFKA_CONTROLLER_QUORUM_VOTERS: 1@kafka:9094
```

## Why a single-node "quorum" is still a quorum

`KAFKA_CONTROLLER_QUORUM_VOTERS: 1@kafka:9094` declares a controller quorum of exactly one voter. Raft elects a leader by majority vote among voters; with one voter, the majority is one, so the node wins instantly and unopposed. There is no contention, no split-brain to resolve — but the *mechanism* is the same one that, with `3` or `5` voters, would run a genuine election and replicate the metadata log across them for fault tolerance.

Pulse runs a single node because this is a local learning cluster, not a production deployment. The election you see above is real; the quorum is just trivially small. Scaling the voter list is the only change needed to make it a multi-node Raft group — no architectural shift.

## The boot delay is why the healthcheck waits

Those few seconds between "container started" and "controller elected" are exactly why Kafka's healthcheck carries `start_period: 20s`. Without it, the probe would mark the container `unhealthy` during a perfectly normal boot. That mechanism is its own concept — see **[Healthchecks](/concepts/healthchecks)**.
