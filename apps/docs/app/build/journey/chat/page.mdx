**Goal:** start the Go gateway, send a chat message from one socket, watch it appear on another — and then prove the parts you can't see in a browser: that the event landed in Kafka **server-authored**, that a channel's messages **co-partition**, that connecting to a channel that isn't live is **rejected**, and that ending a stream **force-closes** its viewers while still emitting their leave.

It's the same installation-lap discipline the [identity journey](/build/journey/identity) used: bring up one capability at a time and read its own proof, so a break tells you exactly which rung failed. The difference is the proofs now live half in the browser (the live push you can see) and half on the log (the durable record you decode with `kcat`).

## Before lights out

chat needs the dev lab healthy — **Kafka** on `:9092` and **Apicurio** on `:8080` — plus the Avro schemas published, and a C toolchain on PATH (`confluent-kafka-go` is cgo, wrapping librdkafka).

```bash
just infra-up                                  # Kafka, Apicurio, Postgres, Redis
bun packages/schemas/scripts/publish.ts        # publish the Avro subjects (idempotent)
xcode-select --install                          # one-time, macOS — the C toolchain for cgo
```

<Callout variant="footgun" title="Wipe Apicurio and chat won't boot — by design">
  Apicurio's dev store is in-memory, so a stack teardown loses the registered
  schemas. chat fetches its three `-value` schemas at startup and **exits at boot
  if any fetch fails** — a gateway that can't encode what it produces shouldn't
  accept connections. The registry is a derived cache; re-publish from the
  `.avsc` source of truth and try again. (Idempotent, so re-running is safe.)
</Callout>

## 1 · Start the gateway

```bash
cd services/chat && go run .
```

Watch for `"msg":"listening","port":"8081"`. The `%4|…GETPID…retrying` line is just librdkafka acquiring its idempotence PID — transient, ignore it. The first run pays the **cgo build tax** (it compiles the C client); later runs are quick.

## 2 · Send a message, decode it off the log

A WebSocket is a thin upgrade over HTTP — `websocat` speaks it. Send one line and close:

```bash
echo 'gg wp' | websocat -n1 ws://localhost:8081/ws/alices-channel
```

The server logs `ws connect` and — if all is well — no `produce failed`. Now read the record back off Kafka, decoded through the registry:

```bash
kcat -C -b localhost:9092 \
  -s value=avro -r http://localhost:8080/apis/ccompat/v7 \
  -t chat.messages.v1 -o beginning -e
```

You get the full **server-authored** record: your `body`, plus the gateway-stamped `messageId` (UUIDv7), `channelId`/`streamId`, `userId`, and `sentAt` (server receipt time). You sent two words; the gateway authored everything else.

## 3 · Prove a channel co-partitions

Every message for one channel must land on one partition — that's what keeps a channel [ordered](/kafka/concepts/partitions-and-ordering). Send a few, then group by key and partition:

```bash
for i in 1 2 3; do echo "alice $i" | websocat -n1 ws://localhost:8081/ws/alices-channel; sleep 0.4; done

kcat -C -b localhost:9092 -t chat.messages.v1 -o beginning -e \
  -f 'partition=%p key=%k\n' | sort | uniq -c
```

Every `alices-channel` message shares one key and one partition. (No `-s/-r` here — printing the raw key needs no registry.)

## 4 · The liveness gate

So far the channel didn't need to be live, because a fresh log folds to an empty live-map and the placeholder path admitted you. To exercise the real gate, bring `identity` up — it produces the lifecycle events chat consumes:

```bash
cd services/identity && just identity-dev      # HTTP :3100, emits stream lifecycle
```

chat replays the lifecycle log at boot and folds it to a live-channel map. Now connect to a channel that **isn't** live and watch it bounce:

```bash
sleep 60 | websocat ws://localhost:8081/ws/somebody-not-live
# → closed with code 1008 — the gate working, not a bug
```

Take alice live, and the same connect is admitted:

```bash
just identity-go-live alices-channel           # produces StreamStarted
```

chat logs `stream started … slug=alices-channel channelId=…`. It learned the channel is live from the log — it never called identity to ask.

<LiveMapFromLog />

<Callout variant="lesson" title="The gate is eventually consistent, on purpose">
  Connect in the sliver between a real go-live and chat consuming the
  `StreamStarted` and you'll be rejected with `1008`, even though the channel is
  genuinely live. The client just retries. That lag is the price of learning
  liveness from the log instead of a synchronous call to identity — the
  [consumer-groups](/kafka/concepts/consumer-groups) page is the why.
</Callout>

## 5 · Presence, and the force-close that proves the wristband

Hold a viewer open (a backgrounded `websocat` has EOF stdin and closes itself, so pipe a `sleep` in to keep the socket alive):

```bash
sleep 60 | websocat ws://localhost:8081/ws/alices-channel &
```

A server-authored `ViewerJoined` lands on `chat.presence.joined.v1` — keyed by `channelId`, on the **same partition** as that channel's messages. Now end the stream from identity while the viewer is still connected:

```bash
just identity-end-stream alices-channel
```

chat logs `force-closing channel … conns=1` and closes the socket with `1001` ("going away"). Re-read `chat.presence.left.v1` and the newest `ViewerLeft` carries the **force-closed** `streamId` — proof it was built from the connection's **wristband**, not a live-map lookup (which, by then, would have found nothing, because chat deletes the map entry *before* it force-closes).

<Callout variant="key" title="Why that last step is the whole design in one move">
  The leave is emitted from identity stamped onto the connection at the gate, not
  re-read at produce time. That's what lets a `ViewerLeft` survive its own
  channel going dark — the [server-authored](/kafka/concepts/server-authored-events)
  invariant and the wristband, working together under a `defer`.
</Callout>

## What you just proved

- **Server-authored:** you supplied a `body`; the gateway authored identity and timing.
- **Co-partitioning:** a channel's messages and presence share a partition, keyed by `channelId`.
- **Learned liveness:** chat gated connections from a map it folded out of the log — no call to identity.
- **Best-effort live, durable log:** the in-memory fan-out pushed to the other tab; the log kept the record. (Run a second node and you'll feel the [single-node limit](/go/concepts/websocket-fanout) Phase 2 fixes.)

One gateway, three topics, and the first time a Pulse service learns another's state purely from the stream. That's Phase 1's chat contract, verifiable on demand.
