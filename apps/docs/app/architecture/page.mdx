Here's the whole system in one breath: Pulse is a **control-plane** — everything that happens _around_ a stream (accounts, follows, chat, presence, analytics) is event-driven, with Kafka as the backbone. The actual video and audio? Those bytes take a different road entirely — the **media plane** — and never touch Kafka. That's why you won't see them in the diagram below.

## System topology

Three services, deliberately written in three different languages so the contracts between them have to be _real_, sit on top of four infrastructure pieces. The diagram is honest about **build state**: the infrastructure runs today, and as of Phase 1 the first service — `identity` — runs with it; `chat` and `analytics` are designed but not yet built.

<SystemTopology />

Hover any node to trace its flows; click one to pin the exact contracts it owns. What the diagram is telling you, read carefully:

- **The four infra nodes are solid — they run today.** `docker compose up` brings Kafka (KRaft), Apicurio, Postgres, and Redis to a healthy state. That was the whole of Phase 0.
- **`identity` is now solid too.** Phase 1's identity-mvp shipped the first real service: a Fastify app on Bun that owns accounts, channels, and streams in Postgres and emits stream-lifecycle events through a transactional outbox. `chat` (Go) and `analytics` (Kotlin) are still designed-not-built — their language is fixed and their events are named in `CONTEXT.md`, but no code produces or consumes yet.
- **`identity`'s edges are solid; the rest stay dashed.** This is the honest part most architecture diagrams hide. `identity` actually drives three flows now — to Postgres, to Kafka, and from Apicurio. Every edge that needs `chat` or `analytics` is still dashed, because those services don't exist yet. Each future epic flips the next node — and its edges — from planned to live.

## The contracts the edges imply

Each edge is a decision already made — click a node above to read its contracts in full. The two flows that run today: `identity → Postgres` together with `identity → Kafka` is the **transactional outbox** — Postgres holds the truth, and events leave as a notification stream rather than the source of record ([read the concept →](/concepts/transactional-outbox)) — while `Apicurio → identity` is **Avro schema resolution**, the registry keeping all three languages agreeing on the shape of every event. The dashed `chat → Redis` is a **read projection** off the chat log — a fast cache for the UI, never the source of truth — still waiting on the `chat` service.
