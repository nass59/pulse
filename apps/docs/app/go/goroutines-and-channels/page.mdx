This is the page that explains why `chat` is written in Go at all. A chat gateway is a pile of long-lived connections, each mostly idle, occasionally chatty — exactly the workload Go's concurrency model was built for.

## Goroutines: threads cheap enough to be free

A **goroutine** is a function running concurrently, scheduled by the Go runtime onto a small pool of OS threads. You start one by putting `go` in front of a call:

```go
go writePump(ctx, conn, send)
```

The magic is the cost. A goroutine starts at a few kilobytes of stack and grows as needed, so having *thousands* alive is normal and cheap — nothing like spawning thousands of OS threads. chat runs one (well, two) per connection without thinking about it. Picture every fan in the stadium following the match on their own at the same time, instead of the venue hiring one staffer per seat.

This is a real departure from Node's single event loop. There's no "don't block the loop" rule to tiptoe around — blocking work just sits in its own goroutine and the scheduler runs everything else.

## Channels: the safe way goroutines talk

If goroutines run independently, how do they share data without a mutex tangle? **Channels** — typed pipes that synchronise a sender and a receiver:

```go
send := make(chan []byte, 16) // a buffered channel of outbound frames

// one goroutine produces:
send <- frame

// another consumes, in a loop:
for frame := range send {
    conn.Write(frame)
}
```

Go's mantra is "don't communicate by sharing memory; share memory by communicating." A channel is a conveyor belt feeding one worker — the producer drops items on, the consumer takes them off, and the language handles the handoff safely. No shared-state scramble, no lock you forgot to release.

## The pattern chat actually uses

Each WebSocket connection gets **two** goroutines: a *read loop* pulling frames off the socket, and a *write pump* draining an outbound channel back to it. They're joined by that channel, and their lifetimes are tied together by a `context`:

<GoroutineReadLoop />

```go
ctx, cancel := context.WithCancel(context.Background())
go writePump(ctx, conn, send) // drains `send`, watches ctx
defer cancel()                 // stop the pump when the read loop returns

for {
    frame, err := conn.Read()
    if err != nil {
        return // triggers the deferred cancel()
    }
    produceAndFanOut(frame)
}
```

<Callout variant="lesson" title="Why context.WithCancel — the goroutine leak it prevents">
  A goroutine that never returns is a leak: per dead connection, a write pump
  would sit forever waiting on a channel nobody sends to, holding memory. Tying
  it to a context means the read loop's `cancel()` makes the pump's{" "}
  `<-ctx.Done()` fire, so it returns cleanly. It's the manager pulling a player
  the instant the whistle goes — nobody's left jogging an empty pitch. This was a
  real fix during the build: the naive version leaked one goroutine per
  disconnect.
</Callout>

<Callout variant="footgun" title="Don't read the request context on a hijacked socket">
  A WebSocket upgrade *hijacks* the underlying connection, so the original
  `http.Request`'s context can fire at the wrong time once the socket is detached.
  chat creates its own `context.WithCancel` for the connection's lifetime rather
  than reusing `r.Context()` — a subtle trap when you're new to both WebSockets
  and Go's context model.
</Callout>
