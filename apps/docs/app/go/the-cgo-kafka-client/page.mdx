chat produces with `confluent-kafka-go`, which is a thin Go layer over **librdkafka** — the reference C client, and the fastest, most feature-complete Kafka library in any language. Choosing it means crossing the boundary from Go into C, and that boundary shows up in three concrete ways worth understanding before they surprise you.

## cgo: Go calling C, and the tax at the door

**cgo** is Go's mechanism for calling C code. Because `confluent-kafka-go` binds to librdkafka through it, building chat isn't pure Go anymore — it needs a **C toolchain** on the machine (`xcode-select --install` on macOS), and the first compile is noticeably slower while it builds the C side.

```bash
# the cgo tax: a C compiler must be present, first build is slow
cd services/chat && go run .
```

That's the trade: you field a world-class import, but you sort the work visa first. For Pulse it's worth it — the alternative pure-Go clients are less complete, and feeling *why* the production choice carries a cost is part of the lesson.

<Callout variant="footgun" title="The startup line that looks like an error but isn't">
  On boot you'll see `%4|…GETPID…retrying` from librdkafka — it's just the client
  acquiring its idempotence producer ID, transient and harmless. Like the
  `TimeoutNegativeWarning` on the identity service, it's noise from a lower layer,
  not a failure. Knowing which scary-looking lines to ignore is half of operating
  a new stack.
</Callout>

## Delivery reports are asynchronous — on a channel

A Kafka producer doesn't block waiting for the broker. `Produce` just *enqueues* the message into librdkafka's internal queue and returns immediately; the confirmation that it actually landed comes back **later**, asynchronously, on a delivery channel you drain in a goroutine:

```go
p.Produce(&kafka.Message{ /* ... */ }, nil)

// elsewhere, a goroutine drains delivery reports:
for e := range producer.Events() {
    if m, ok := e.(*kafka.Message); ok && m.TopicPartition.Error != nil {
        slog.Error("produce failed", "err", m.TopicPartition.Error)
    }
}
```

If that shape looks familiar, it should — it's the exact [goroutine-plus-channel](/go/goroutines-and-channels) pattern from the last page, now applied to delivery confirmations. You hand the parcel to the courier and get the "delivered" notification afterwards, rather than standing at the door until it arrives. On shutdown you `Flush` to make sure the queue drains before the process exits.

## The Avro struct-tag gotcha

This one cost real debugging time. chat encodes events as Avro, mapping Go struct fields to schema field names with **struct tags**:

```go
type ChatMessageSent struct {
    MessageID string `avro:"messageId"` // tag, not the field name, is the wire name
    Body      string `avro:"body"`
}
```

Go's naming convention capitalises initialisms — the field is `MessageID`. But the Avro schema's field is `messageId`. The bridge between them is the `avro` struct tag, and **its casing must match the schema exactly**. Write `avro:"messageID"` and the field silently serialises under the wrong name — no error, just data that doesn't match the contract. It's spelling a teammate's name wrong on the team sheet: looks fine to you, the system can't match it.

<Callout variant="lesson" title="The compiler can't catch a wrong tag">
  Struct tags are just strings — the Go compiler never validates them. A casing
  typo passes `go build` happily and only shows up when you decode the event and a
  field is missing. The defence is to decode your own output once with `kcat`
  against the registry (the [chat journey](/journey/chat) does exactly this) and
  eyeball that every field is present and correctly named.
</Callout>

That's the Go tour for reading `chat`: errors as values, goroutines and channels for concurrency, and a C-backed client with a few sharp edges. Pair it with the [Concepts](/concepts) tier — the Go is *how*, the Kafka concepts are *why* — and the gateway reads end to end.
