## The idea: publish once, let everyone read

Kafka flips it around. Instead of services calling each other, a producer **publishes an event to a log** ‚Äî "this like happened" ‚Äî and any number of consumers read from that log, each at its own pace. Add a new feature next quarter? It just starts reading the log. Nobody who already publishes has to change a line.

<Callout variant="key" title="Almost everything is an event">
  An *event* is just a fact that already happened, written down: `PhotoLiked`,
  `StreamStarted`, `PaymentCaptured`. Once you start seeing your system as a
  river of events rather than a web of function calls, Kafka stops looking like
  a queue and starts looking like the obvious place to put the river.
</Callout>

So what *is* Kafka? Three things, together:

- **A log** ‚Äî an append-only, ordered, durable record of events. Writes go on the end; nothing gets edited in place.
- **A broker** (usually several) ‚Äî the servers that store the log and hand pieces of it to whoever asks.
- **Clients** ‚Äî *producers* that append events and *consumers* that read them. They never talk to each other, only to the log.

That's the whole shape. A producer drops a fact on the end of the log; consumers read it whenever they're ready; the log keeps it around long after it's been read, so a brand-new consumer can show up tomorrow and replay everything from the beginning.

## Why a log, and not a queue?

A traditional message queue *delivers and forgets* ‚Äî once a worker takes a message, it's gone. That's fine until a second team wants the same messages, or you need to reprocess last week's data after a bug fix. A log keeps every event for a configured window, and every consumer tracks *its own* position. One team's reading never consumes another team's copy. That single property ‚Äî **retain, don't delete on read** ‚Äî is what makes Kafka the backbone for analytics, search indexes, caches, and audit trails all reading the same stream at once.
