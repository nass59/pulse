Reading about a log only gets you so far. In ten minutes you'll have a real broker running and your first message round-tripped — no service code, no schemas, just proof the thing works. This is exactly how Pulse starts: Phase 0 stands up the broker and verifies it before a single line of application code exists.

## 1 · Bring the broker up

Pulse runs its local Kafka in **KRaft mode** — Kafka with no ZooKeeper, the broker electing itself controller in-process. It's defined in `infra/docker-compose.yaml`, so one command brings it online:

```bash
# from the repo root
docker compose -f infra/docker-compose.yaml up -d

# or, via the project's justfile
just infra-up
```

Give it a few seconds, then check it's healthy:

```bash
docker compose -f infra/docker-compose.yaml ps
```

<Callout variant="lesson" title="“Started” isn't “ready”">
  The broker reports `unhealthy` for the first ~15 seconds — that's normal. A
  single-node KRaft broker has to elect itself controller before it can serve,
  and the healthcheck's `start_period` shields those early failures on purpose.
  Two of Pulse's first concepts dig into exactly this:
  [KRaft mode](/kafka/concepts/kraft-mode) and [Healthchecks](/build/architecture/healthchecks).
</Callout>

## 2 · Round-trip a message with kcat

`kcat` is the universal Kafka REPL — a tiny client with no app logic, no schemas, the smallest possible thing that can talk to a broker. Install it (`brew install kcat`), then open two terminals.

Terminal A — **produce** a single record:

```bash
echo "hello pulse" | kcat -P -b localhost:9092 -t pulse.smoke.test
```

Terminal B — **consume** it back from the start:

```bash
kcat -C -b localhost:9092 -t pulse.smoke.test -o beginning -e
```

If `hello pulse` prints in Terminal B, your substrate is real.

## 3 · Read every flag

That one line is the whole produce/consume model in miniature. Here's what each piece does:

| Flag | Means | Why it matters |
|---|---|---|
| `-P` / `-C` | **P**roduce or **C**onsume mode | kcat reads stdin in `-P` (one line → one record); streams the topic in `-C` |
| `-b localhost:9092` | the **bootstrap** broker | first contact only — the broker replies with the real *advertised* address to actually send to |
| `-t pulse.smoke.test` | the **topic** | the dotted name is convention, not hierarchy — Kafka sees one flat string |
| `-o beginning` | start **offset** | `beginning` replays from offset 0; drop it and you only see new records |
| `-e` | **exit** at end of partition | makes `-C` a one-shot read instead of tailing forever |
| `-K:` | **key** delimiter | absent here, so every record had a `null` key — fine with one partition, essential once you want per-key ordering |

<Callout variant="footgun" title="The topic created itself — and that bites in production">
  `pulse.smoke.test` didn't exist; producing to it silently **auto-created** it.
  Handy on a laptop, a real hazard in production where a typo'd topic name
  spawns a phantom topic no consumer reads. Pulse keeps auto-creation on locally
  and treats it as a known footgun — see
  [Auto topic creation](/kafka/concepts/auto-topic-creation).
</Callout>
