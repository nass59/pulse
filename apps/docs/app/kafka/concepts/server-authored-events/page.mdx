Here's a rule that quietly prevents a whole class of bugs and abuse: **everything chat produces is server-authored.** The client never names the channel or stream, never supplies a `userId`, never sets a timestamp. Its entire contribution to a chat message is the text. The gateway stamps the rest.

<ServerAuthoredStamp />

The viewer speaks into a microphone; the gateway is the official scorer who decides what actually goes in the record. That split is the difference between an event you can *trust* and a claim you have to *verify*.

## Why not let the client send the whole event?

Because then anyone could lie. If the browser supplied `userId`, a viewer could post as someone else. If it supplied `sentAt`, a message could be backdated to jump the order or dodge a moderation window. Trusting client-supplied identity or timing means validating all of it server-side anyway — so you may as well author it server-side in the first place and make forgery *impossible by construction* rather than merely *detected*.

So the gateway stamps:

- **`messageId`** — a server-minted UUIDv7 (time-ordered, so ids sort roughly by creation).
- **`userId`** — the account on the connection, never a value from the frame.
- **`channelId` + `streamId`** — the channel and stream this socket joined.
- **`sentAt`** — the gateway's own receipt time.

<Callout variant="key" title="One invariant, every topic chat owns">
  This isn't a chat-message quirk — it's chat-wide. Presence events
  (`ViewerJoined` / `ViewerLeft`) are server-authored too: the client's only
  "input" is the act of opening or closing a gated socket. The gateway stamps the
  identity and timing on those exactly the same way. Authorship can't be forged
  anywhere chat produces.
</Callout>

## The wristband: identity captured at the gate

Where do `channelId` and `streamId` come from on every message? Not a per-message lookup in the live-channel map — that would be wasteful and could drift if the stream ended mid-session. Instead, when the socket connects and passes the [liveness gate](/kafka/concepts/consumer-groups), the gateway pins that channel and stream **onto the connection itself**. We call it the **wristband**: you're checked once at the gate, given the band, and it's read all night.

```text
join  → gate passes → stamp {channelId, streamId} onto the conn   (the wristband)
msg   → read body from the frame, read identity from the conn      (not a lookup)
produce → ChatMessageSent { body, ...wristband, userId, sentAt }
```

Every later message reads the wristband off the connection — fast, and stable for the life of the session. Even when a stream ends and chat force-closes the socket, the final `ViewerLeft` is built from that same wristband, so it carries the correct `streamId` it was always associated with.

## Receipt time is what orders a channel

Because `sentAt` is always the gateway's clock — never a client's — there's a single authority timing every message in a channel. Client clocks are wrong, skewed across timezones, or outright spoofable; leaning on them would scramble order. With one server clock stamping every record and [per-channel partitioning](/kafka/concepts/partitions-and-ordering) keeping them in one lane, a channel's messages are totally ordered *as the gateway witnessed them*. It's the finish-line camera deciding the race, not each runner's wristwatch.
