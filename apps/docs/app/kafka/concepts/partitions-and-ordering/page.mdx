A single log on one machine would be a bottleneck, so Kafka splits every topic into **partitions** — independent sub-logs it can spread across the cluster and hand to different consumers in parallel. Partitions are the **unit of parallelism**. But they're also the **unit of ordering**, and that second job is where all the interesting decisions live.

The rule is short and absolute: **Kafka guarantees order within a partition, never across partitions.** Picture supermarket checkout lanes. People in one lane are strictly in order; there's no meaningful order *between* lanes. So the only question that matters is: *which records land in the same lane?*

## Same key, same lane

A producer attaches a **key** to each record. Kafka hashes that key (Pulse uses `murmur2`, the kafkajs/librdkafka default) modulo the partition count to pick the partition. Same key → same partition → one ordered lane. chat keys every message by `channelId`:

<CoPartitionRouting />

That's the whole trick. Every message in alice's channel shares her `channelId`, so they all queue in a single lane and come out in the order the gateway received them. Key by something random — a `messageId`, say — and alice's own messages would scatter across all six partitions and could surface out of order. One line of key choice is the difference between coherent chat and nonsense.

<Callout variant="key" title="You don't need global order — you need per-channel order">
  Total ordering across a whole topic would mean a single partition, which means
  a single consumer and zero parallelism. Pulse doesn't want that. It wants each
  *channel* ordered and all channels processed in parallel — which is exactly
  what keying by `channelId` over many partitions delivers.
</Callout>

## Co-partitioning: the quiet superpower

Look again at the figure: a message, a presence join, and a lifecycle event — **three different topics** — all keyed by the same `channelId`. That means alice's records land on the **same partition number** in every one of those topics. This is **co-partitioning**, and it's deliberate (ADR-recorded, keyed and provisioned identically across chat and the stream-lifecycle topics).

Why care? Because `analytics` will later need to join a channel's chat stream against its stream-state — "how many messages per minute *while this stream was live*". If the related records already sit together on the same partition, that join happens locally, with no expensive **repartition** shuffle first. It's filing one club's fixtures, results, and squad sheet under the same folder so you can cross-reference without re-sorting the whole league.

## The partition count is a contract

Here's the trap that makes people treat partition count carefully. The key→partition mapping is `hash(key) mod partitionCount`. Change the count and the modulo changes, so existing keys rehash to *new* partitions — while their old records stay where they were.

<PartitionContractTrap />

For a keyed topic that's not a tuning knob you nudge later; it's a migration. A channel's ordered lane splits at the seam: old messages on the old partition, new ones somewhere else. That's why Pulse provisions every keyed topic at its contracted **6 partitions** up front instead of accepting the broker's 1-partition default — re-seeding the bracket at half-time isn't allowed once the games have started.
